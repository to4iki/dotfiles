# Commit

現在のGitワークスペースの状態を詳細に分析し、適切なファイルをステージングした上で、構造化されたコミットメッセージとともにコミットを実行する。

## ルール

- **セキュリティ第一**: `.env` ファイルや、認証情報、トークン、機密情報が含まれるファイルは、いかなる場合もステージングおよびコミットの対象にしない
- **メッセージフォーマット**: Conventional Commits 仕様を厳格に遵守する
- **言語**: コミットメッセージは英語で記述する。ユーザーから明示的に日本語での記述を指示された場合のみ日本語を使用する
- **原子性**: 1つのコミットは1つの論理的な変更単位とする。複数の無関係な変更がある場合は、それぞれ別コミットに分けることをユーザーに提案する
- **追跡可能性**: Issue番号やタスク番号があればコミットメッセージ本文に含める

## Conventional Commits タイプ

| タイプ | 用途 |
|--------|------|
| `feat` | 新機能の追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（フォーマット等） |
| `refactor` | バグ修正でも機能追加でもないコード変更 |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセスや補助ツールの変更 |
| `ci` | CI設定の変更 |
| `perf` | パフォーマンス改善 |

スコープは任意。影響範囲が明確な場合は `feat(auth):` のように付与する。

## 実行手順

1. **変更内容の分析**: 以下のコマンドを並列で実行し、変更の意図と影響範囲を把握する
    - `git status` - 未追跡ファイルと変更ファイルを確認
    - `git diff` - 未ステージの変更内容を確認
    - `git diff --staged` - ステージ済みの変更を確認

2. **ステージング**: 機密ファイルを除き、コミット対象のファイルをステージングする
    - `git add <files>` でファイルを個別指定する（`git add .` は避け、意図的に選択する）

3. **コミット実行**: 変更の「理由」に焦点を当てたメッセージでコミットする
    ```bash
    git commit -m "$(cat <<'EOF'
    <type>(<scope>): <subject>

    <brief explanation of why (optional)>

    Co-Authored-By: Claude Haiku 4.5 <noreply@anthropic.com>
    EOF
    )"
    ```
    > **重要**: HEREDOC の `EOF` マーカーは行頭（インデントなし）に置くこと。インデントがあると heredoc が正しく機能しない。

4. **確認**: `git status` を実行し、コミットが正常に完了したことを確認する

## トラブルシューティング

- コミットすべき変更が一切存在しない場合は、処理を中断し、その旨をユーザーに通知すること
- Git hook（例: pre-commit など）が失敗した場合は、処理を中断し、ユーザーにエラーの解消を依頼すること
